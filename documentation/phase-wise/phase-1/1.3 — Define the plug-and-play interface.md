
- write(df, table_path, key_cols) — insert or upsert records
- read(spark, table_path) — read the full table as a DataFrame
- delete(spark, table_path, condition) — delete rows matching a SQL condition
- get_history(spark, table_path) — return the transaction/commit history

Q. Can you explain the operations of each of the function in the interface?     
A. 

Your orders table will look something like this:

| order_id | status | amount | updated_at |
|---|---|---|---|
| ORD-0001 | created | 450 | 2024-01-01 10:00 |
| ORD-0002 | created | 230 | 2024-01-01 10:01 |
| ORD-0003 | created | 890 | 2024-01-01 10:02 |

Now let's walk through each operation.

---

## 1. `write(df, table_path, key_cols)`

This is not a simple append. It's an **upsert** — a combination of insert and update.

The `key_cols` argument is the important one. It tells the client *"this is what makes a record unique"*. In your case it's `order_id`.

**Scenario A — Record doesn't exist yet (INSERT)**
A new order `ORD-0004` arrives. The client checks — no existing record with `order_id = ORD-0004`. So it inserts it as a new row.

**Scenario B — Record already exists (UPDATE)**
`ORD-0001` comes in again with `status = paid`. The client finds an existing row with `order_id = ORD-0001` and updates it in place rather than adding a duplicate row.

So after processing an update event your table becomes:

| order_id | status | amount | updated_at |
|---|---|---|---|
| ORD-0001 | **paid** | 450 | **2024-01-01 10:15** |
| ORD-0002 | created | 230 | 2024-01-01 10:01 |
| ORD-0003 | created | 890 | 2024-01-01 10:02 |

This is exactly why plain Parquet files on HDFS were painful — they had no concept of upsert. You'd have to rewrite the entire file. Hudi/Delta/Iceberg handle this for you behind the scenes.

---

## 2. `read(spark, table_path)`

This is the straightforward one. It reads the **current snapshot** of the table and returns it as a Spark DataFrame.

The interesting thing is what "current snapshot" means in the context of table formats. Because Hudi/Delta/Iceberg maintain a transaction log, when you call `read` you always get a **consistent view** — you never see a half-written batch or a partially updated row. It's like a database read, not a raw file read.

In practice you use this to:
- Query the current state of all orders
- Feed data into a Spark aggregation or report
- Verify what was written after a `write` call

---

## 3. `delete(spark, table_path, condition)`

Deletes all rows matching a SQL condition string. For example:

```
condition = "order_id = 'ORD-0002'"
condition = "status = 'cancelled'"
condition = "amount < 100"
```

The reason this is significant — again going back to the problem with plain Parquet — is that **you cannot delete a row from a Parquet file**. Parquet is immutable once written. Hudi/Delta/Iceberg solve this by writing a **delete marker** into the transaction log, and the next time you read the table those rows are excluded.

The real-world trigger for this in your e-commerce domain is **GDPR right-to-erasure** — a user requests their data be deleted. Without a table format, you'd have to rewrite every Parquet file that contains their data. With the table format, it's a single delete operation.

---

## 4. `get_history(spark, table_path)`

This returns the **transaction log** of the table — every operation that was ever performed on it, in order. Something like:

| version | timestamp | operation | details |
|---|---|---|---|
| 0 | 10:00:00 | CREATE | Initial write, 3 rows |
| 1 | 10:15:00 | MERGE | Upserted 1 row (ORD-0001 → paid) |
| 2 | 10:30:00 | DELETE | Deleted ORD-0002 |

This unlocks two powerful capabilities:

**Time travel** — because every version is logged, you can read the table *as it was at any point in the past*. "Show me what the orders table looked like at 10:05am" — before the update and delete happened. This is invaluable for debugging pipelines and auditing.

**Incremental reads** — instead of reading the whole table every time, you can say "give me only the rows that changed since version 1". This is what makes Hudi particularly good for streaming — Flink can ask "what changed since I last checked?" rather than scanning everything.

---

## How They Relate to Each Other

```
write()  →  creates versions in the log
read()   →  always reads the latest version
delete() →  creates a new version marking rows as deleted
history()→  lets you see and navigate all versions
```

The transaction log is the thread that connects all four operations. Without it, you just have files. With it, you have something that behaves like a database.
